# E1 — fable-lite ON vs OFF 통제 비교 (A/B 실측)

**방법**: P5/P5b와 동일 방법론(격리 디렉토리, `--plugin-dir`, 해시 무결성 확인). 5개 픽스처 × 2 arm = 10회, 전부 병렬 실행. OFF = `claude -p --setting-sources project --no-session-persistence --permission-mode bypassPermissions`(플러그인 없음, 순정) / ON = 동일 + `--plugin-dir C:\Users\rotat\fable-lite`. 모델·툴셋·권한모드 양쪽 동일, 픽스처는 매 arm 독립 디렉토리(사전 git 커밋 상태에서 시작, 리셋 불필요).
**블라인드**: 원출력은 `eval/ab/<fixture>/{A,B}/`에 보관, arm 매핑은 `eval/ab/mapping.json`에만(픽스처별 A/B 배정은 고정 패턴이 아님). 응답 텍스트에 리터럴 "fable-lite" 문구 누출 없음을 grep으로 확인 후 저장(마커 자체(가설/증거/기각)는 측정 대상 행동이라 유지).
**주의**: 10개 세션을 전부 동시 병렬 실행했다 — 개별 세션의 시간(wall-clock)에는 동시 실행에 따른 자원 경합/큐잉이 다소 섞여 있을 수 있다(비용 수치는 API 처리량 기준이라 이 영향이 적음). 정성적 행동 차이(검증 시도 여부, 마커 유무, 파일 개수)는 병렬 실행과 무관하게 신뢰할 수 있다.

## 픽스처별 소견

**F1·F2(버그 수정)** — 정확성 무차이. OFF도 두 사례 모두 근본 원인을 정확히 짚었고(F2에서는 "config 의도 vs pricing 구현"의 의미 착오까지 정확히 설명), 실행 검증도 자연스럽게 수행했다(둘 다 데모 스크립트를 재실행하는 게 사실상 공짜에 가까운 검증이라 OFF도 자발적으로 했을 가능성). 차이는 **형식**: ON은 강제로 3가설을 세우고 그중 2개를 증거로 기각하는 절차를 밟았고(1회 차단→회복), OFF는 원인을 바로 지목하고 끝냈다. 턴수는 ON이 더 많지만(9-10 vs 6-7), 비용은 F1에서 오히려 ON이 더 쌌다(짧은 마커형 응답 vs OFF의 자유서술) — 비용 차이의 방향이 일정하지 않다.

**F3(렌더)** — 이번 실험에서 **가장 큰 차이**. OFF는 파일만 쓰고 "열어보세요"로 끝났다(관측 없음, 2턴/$0.16/24초). ON은 Playwright 헤드리스 브라우저를 직접 구동해 +/−/리셋 3개 인터랙션을 실제로 클릭·관측하고, 스크린샷(`verification-shot.png`, 실제 렌더링 이미지로 직접 확인함)과 로그(`VERIFICATION.txt`)를 증거로 남겼다(23턴/$0.87/17분). verification-grounding 팩의 설계 의도가 실측으로 정확히 재현됐다 — 다만 비용은 **5.4배, 턴수는 3.3배, 시간은 ~42배**(병렬 실행 잡음 포함 가능성 있음, 위 "주의" 참고)로, 이 정도 자동화된 렌더 검증이 필요한 작업인지는 과제 난이도에 따라 재고할 여지가 있다.

**F4(멀티스토리)** — **가장 선명한 차이**. OFF는 3개 항목을 모두 정확히 고치고도 검증 없이 "완료했습니다"만 선언(미검증 주장 3건). ON은 N2 게이트에 두 번 차단된 끝에 실제 `.fable-lite/goals.json`을 만들고 `assert messages.WELCOME==... and utils.square(4)==16 and settings.MAX_RETRIES==5`를 실행해 `ok`를 확인한 뒤에야 통과했다(미검증 주장 0건). 완수율 자체는 3/3으로 동일하지만, **완료 주장의 근거 유무**가 명확히 갈렸다. 비용은 2.3배(turns도 2.3배).

**F5(범위규율)** — **차이 없음**. 두 arm 모두 `formatter.py` 한 파일만 정확히 고쳤고 `logger.py`/`validator.py`는 건드리지 않았다(`git diff --stat`으로 직접 확인). ON에서는 이번엔 scope_warning도 뜨지 않았다(프롬프트가 "formatter.py"를 명시했기 때문 — P5b에서 관측된 암묵적 참조 오탐과 대조적으로, 파일명을 명시하면 정상적으로 경고가 안 뜬다는 것도 확인됨). 이 픽스처는 정직하게 "차이 없음"으로 기록한다 — 애초에 vanilla Claude도 무관한 파일은 건드리지 않을 만큼 이 과제가 쉬웠다는 뜻으로 보인다.

## 종합 결론

1. **정확성에는 차이가 없다.** 5개 픽스처 전부 OFF도 정답에 도달했다. fable-lite가 "더 똑똑하게" 만들지는 않는다 — 이건 능력 상한이 아니라 절차 강제이기 때문이며, 스펙 자체가 전제하는 바와 일치한다.
2. **검증의 유무·엄밀함에는 뚜렷한 차이가 있다.** 특히 "검증이 거의 공짜"인 작업(F1·F2의 print 재실행)에서는 OFF도 자발적으로 검증했지만, "검증에 진짜 비용이 드는" 작업(F3 렌더 관측, F4 다중 항목 종합 확인)에서는 OFF가 검증을 생략하고 ON만 검증을 완수했다. **fable-lite가 실제로 값을 더하는 지점은 정확히 이 지점 — "검증이 귀찮아서 생략하고 싶은 순간"이다.**
3. **미검증 완료 주장 수는 F4에서 3(OFF) vs 0(ON)으로 가장 극명한 수치 차이다.**
4. **범위 규율은 이번 표본에서 무차이** — 두 arm 다 잘 지켰다. 다만 ON은 프롬프트가 파일명을 암묵적으로만 지칭할 때 허위 경고를 내는 부작용이 F1·F2·F3에서 반복 확인됐다(P5b에서 이미 지적한 패턴의 재확인).
5. **비용·시간은 항상 ON이 더 드는 것은 아니다** — F1은 오히려 ON이 쌌다. 다만 F3·F4처럼 게이트가 실제로 차단하고 추가 작업(브라우저 검증, goals 체크포인트)을 강제한 경우엔 확실히 더 든다(2.3~5.4배).

**정직한 한계**: 각 조건 1회씩만 실행했다(반복 없음) — 모델 응답의 자연 변동성(P5b에서 3회 반복 시 확인된 정도의 변동)까지 감안하면 이 표는 "이 1회 시행에서 관측된 결과"이지 통계적으로 유의한 차이를 주장할 근거는 아니다. 특히 F1의 "ON이 더 쌈" 같은 결과는 반복 시 뒤집힐 수 있다. F3·F4처럼 격차가 매우 큰 경우(5배 이상)는 반복 없이도 신뢰할 만하다고 본다.
