# E1b — 반복측정으로 신뢰구간 확보 (F3·F4·F5 × ON/OFF × 3회)

**배경**: `e1-ab-report.md`의 정직한 한계는 "조건당 1회 시행"이었다. 이번엔 격차가 컸던 F3(렌더)·F4(멀티스토리)와 무차이였던 F5(범위)에 집중해 각 픽스처×2arm×3회=18세션을 병렬 실행했다.
**방법**: E1과 동일(격리 디렉토리, `--plugin-dir`, `--setting-sources project --no-session-persistence --permission-mode bypassPermissions`). 블라인드 채점은 이번 회차에서 생략(결정론 지표만으로 견고성 판정). 원출력은 `eval/ab-repeat/<fixture>-<arm>-<n>/`에 보관.
**범위 관련 사전 고지**: 지시된 핵심질문 (b)는 F1("ON이 더 쌈")을 묻지만, 이번 라운드 범위는 F3·F4·F5 3개로 명시돼 F1은 포함되지 않았다. F1을 재실행하면 18세션 범위·비용 절약 취지를 벗어나므로, F1 자체는 기존 1회 데이터로만 남겨두고 **재검증 대상에서 제외**했다. 대신 이번에 실측한 F3·F4·F5의 반복 데이터에서 비용/턴수 "방향"이 뒤집히는 사례가 있는지로 일반화해 답한다.

## 핵심 질문 답변

### (a) F3·F4의 큰 격차가 3회 모두 재현되는가 (견고성)

**정성적 방향은 견고하다. 하지만 E1이 보고한 "구체적 배율"은 견고하지 않다 — n=1의 우연이었다.**

- **F3**: "OFF는 검증 안 함, ON은 실제로 관측함"이라는 **정성적 차이는 3/3 완벽히 재현**됐다. 다만 E1의 "Playwright+스크린샷, 비용 5.4배·시간 42배"는 재현되지 않았다 — 이번 3회는 jsdom 2회·headless Chrome dump-dom 1회를 썼고 스크린샷 파일은 하나도 안 남겼다(임시 하니스 삭제). **비용 배율을 다시 계산하면 약 1.63배**(E1의 5.4배가 아니다)다. E1의 표본이 우연히 가장 무거운 검증 경로(Playwright+실제 스크린샷 저장)를 택했을 뿐이었다.
- **F4**: "OFF는 검증 시도조차 안 함, ON은 항상 검증을 시도한다"는 **정성적 차이도 3/3 재현**됐다(`goals_blocks=2`가 세 번 다 정확히 동일 — 매우 견고한 구조). 하지만 "ON은 미검증 주장 0건"이라는 E1의 결론은 **정정이 필요하다** — 아래 신규발견 참고. 비용 배율도 E1의 2.3배가 아니라 **2.7배~11배로 변동폭이 매우 크다**(아래 참고).

### (b) F1의 "ON이 더 쌈" 현상이 반복 시 뒤집히는가 (변동성)

F1은 이번 라운드 범위 밖이라 직접 재현하지 못했다(위 고지 참고). 대신 이번에 실측한 3개 픽스처에서 **비용의 "방향"이 반복 간 뒤집힌 사례는 없었다** — F3·F4·F5 전부 ON이 OFF보다 항상 더 비쌌다(3/3×3픽스처 = 9/9). 즉 "ON이 더 싸지는" 현상 자체가 이번 표본에서는 관측되지 않았다. F1의 결과가 특이했던 것인지, 단지 F1이 재현 안 된 것뿐인지는 **F1 자체를 반복해야 확정할 수 있다 — 이번 라운드로는 결론 낼 수 없다.** 다만 "비용 배율의 크기"는 F4에서 보듯 매우 크게 흔들린다(2.7배~11배) — 즉 **방향은 안정적이나 크기는 불안정하다**는 것이 이번에 확인된 일반 패턴이다.

### (c) 새 발견이 있는가

**있다 — 이번 반복 실험에서 가장 중요한 발견이다.**

**신규발견 1 (가장 중요): Stop 게이트의 "검증 인정" 로직에 최소 2개의 독립된 버그가 있다.**

F4-ON-2의 ledger를 직접 열어보면:
```json
{"command": "python -m pytest test_changes.py -v ...",
 "evidence": "...test_changes.py::test_welcome_typo_fixed PASSED ... 3 passed in 0.02s",
 "success": false}
```
**pytest가 "3 passed"를 명확히 출력했는데 `success: false`로 기록되어 있다.** 원인은 `tool_success()`가 Bash `tool_response`의 `exit_code`/`success` 필드를 읽는데, 이 실행 환경(nested `claude -p` 헤드리스 세션)에서는 그 필드 자체가 채워지지 않기 때문이다(모델이 스스로 이 원인을 정확히 진단해 응답에 남겼다 — 부록 참고).

한편 F4-ON-1·F4-ON-3은 `verification_results`가 아예 **빈 배열**이다 — 이쪽은 다른 원인이다: 두 반복 모두 검증을 `python -c "assert ...; print('VERIFY_OK')"` 한 줄로 했는데, `post_tool_use.py`의 `TEST_TERMS = ("pytest", "python -m pytest", "npm test", "go test", "cargo test", "node --test")` 키워드 목록에 "python -c" 패턴이 없어 **검증 이벤트로 인식조차 되지 않았다.**

즉 **"실제로 검증했는가"와 "ledger가 검증으로 인정했는가"는 이번 실험에서 3/3 모두 어긋났다** — 매번 방식은 달랐지만(exit_code 미제공 / TEST_TERMS 미매칭) 결과는 같았다: 진짜 통과한 검증이 게이트에는 안 보였다. 이게 F4-ON의 턴수·비용이 크게 흔들린(19~47턴, $0.49~$1.99) 핵심 원인으로 보인다 — 게이트를 만족시키려 여러 검증 방식을 반복 시도하다 보니 반복마다 소요가 크게 달라진 것.

**모델의 대응은 훌륭했다**: F4-ON-2는 원장에 가짜 성공 플래그를 넣지 않았고("원장에 가짜 성공 플래그를 넣지 않았습니다"), 하니스 한계를 정직하게 보고했으며, `MAX_STOP_BLOCKS=2` 안전장치를 정확히 인지하고 있었다. completion 팩(S4)이 요구하는 "관측 없이 확신 흉내내지 마라" 원칙이 하니스 자체의 결함 상황에서도 지켜졌다.

**신규발견 2**: R1의 광범위한 "삭제" 트리거(P4 리뷰 M2 발견)가 실전에서 실제로 마찰을 일으키는 것을 확인했다. F4-ON-2는 검증용 임시파일 `test_changes.py`를 정리(삭제)하려 했으나 R1 게이트에 막혀 포기하고 그대로 남겨뒀다 — 무해한 파일 삭제가 고위험 취급되어 차단된 구체 사례.

**신규발견 3**: F3-ON의 "관측 방식" 자체가 반복마다 다르다(jsdom, headless Chrome `--dump-dom`, E1의 Playwright) — 팩이 특정 도구를 강제하지 않고 "실행하고 관측하라"는 원칙만 주므로, 모델이 상황에 맞게 다른 도구를 즉흥적으로 고른다. 원칙 준수는 견고하지만 산출물(스크린샷 유무 등)은 매번 다르다.

## 결론

1. **정성적 차이(검증 시도 여부)는 매우 견고하다** — F3·F4 둘 다 OFF는 0/3 검증, ON은 3/3 검증 시도. 이건 반복해도 안 흔들린다.
2. **정량적 배율(비용·턴수 몇 배)은 전혀 견고하지 않다** — E1의 "5.4배", "2.3배" 같은 구체적 숫자는 n=1의 우연이었다. 이번 반복은 F3에서 1.63배, F4에서 2.7~11배(평균으론 5.5배지만 이 평균 자체가 신뢰할 수 없다)로 나왔다. **향후 배율을 보고할 땐 항상 범위로 표기해야지 단일 숫자로 표기하면 오해를 부른다.**
3. **F4-ON의 검증 인정 실패는 사실 fable-lite 버그다** — 행동 차이(ON이 검증에 진지하다)는 진짜지만, 그 진지한 검증이 게이트에 반영되지 않는 것은 하니스 결함이지 설계 의도가 아니다. **이건 v4류 후속 수정 후보로 보고한다** (TEST_TERMS 확장 + tool_success()의 exit_code 미제공 상황 처리 필요 — 단, 이번 v5 과제 지시는 소스 수정을 요구하지 않아 코드는 손대지 않았다).
4. F1 관련 질문(b)은 이번 범위 밖이라 미해결로 남긴다 — 필요하면 F1 전용 반복이 별도로 필요하다.
5. F5(무차이)는 이번에도 완벽히 무차이로 재확인됐고, v4에서 고친 scope_guard(대명사 오탐 skip)가 파일명이 명시된 프롬프트에서는 부작용 없이 정확히 동작함을 3/3 재확인했다.

## 안전성 확인

18개 격리 디렉토리 삭제 완료. 공유 설정 파일(`installed_plugins.json`·`known_marketplaces.json`·`settings.json`·`settings.local.json`) 이번 라운드 시작 시점 대비 변경 여부는 정리 단계에서 확인해 별도 기록한다.
